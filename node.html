Overview

Why Run a Sonic Node?

Running a Sonic node gives you direct, trustless access to the network while contributing to its decentralization and security.

Running your own Sonic node is the most direct and trustless way to interact with the Sonic network. Rather than relying on third-party RPC providers, a self-hosted node gives you your own private JSON-RPC endpoint with no rate limits, no data sharing with external parties, and the assurance that you are reading the canonical blockchain state directly. For serious dApp developers, protocol teams, and trading operations, a self-hosted node is often preferable to shared RPC infrastructure.

Beyond direct access, running a Sonic node contributes to the network's decentralization and censorship resistance. The more geographically distributed and independently operated full nodes exist in the network, the more resilient Sonic is to coordinated attacks, infrastructure failures, or attempts to censor transactions. Every additional node strengthens the collective security and availability of the network for all participants.

🏃
Full Node
A full node verifies every transaction and maintains the current state of the blockchain. It provides a private JSON-RPC endpoint and participates in gossip propagation. Full nodes do not vote in consensus and do not require staked tokens. Ideal for developers and infrastructure providers.
🗳️
Validator Node
A validator node participates in Lachesis aBFT consensus by creating and voting on event blocks. Validators must stake S tokens and are rewarded with staking income and FeeM participation. Running a validator requires high uptime and reliable hardware to avoid slashing penalties.
📚
Archive Node
An archive node retains the complete historical state of the blockchain at every block height, not just the current state. Required for serving historical eth_call queries and powering block explorers, analytics platforms, and compliance tools that need deep historical data.

Thanks to SonicDB's 96% reduction in storage requirements, Sonic nodes are dramatically more economical to operate than comparable nodes on other EVM chains. An Ethereum archive node can require 10+ TB of NVMe storage; a Sonic full node running SonicDB operates comfortably on a fraction of that. This makes it realistic for individual developers, small teams, and community members to run their own nodes on consumer hardware or affordable cloud VPS instances, broadening the node operator base beyond well-funded institutions.

Hardware

Hardware Requirements

SonicDB's efficiency means Sonic nodes can run on significantly less hardware than competing L1 nodes of equivalent capability.

Full Node — Minimum Specifications

CPU4 cores, 3.0 GHz+
RAM16 GB DDR4
Storage500 GB SSD (NVMe recommended)
Network100 Mbps stable connection
OSUbuntu 22.04 LTS or Debian 12

Validator Node — Recommended Specifications

CPU8+ cores, 3.5 GHz+ (AMD EPYC or Intel Xeon)
RAM32 GB DDR4 ECC
Storage1 TB NVMe SSD (high IOPS)
Network1 Gbps with <10ms latency to peer validators
RedundancyUPS backup power, RAID storage
OSUbuntu 22.04 LTS (server edition)

The SonicDB storage architecture separates hot state (current account balances and contract storage values, accessed on every transaction) from warm state (recently modified data) and cold/archive state (historical data rarely needed during normal operation). This tiered access pattern means that the NVMe SSD requirement applies primarily to the hot state cache, which is relatively small compared to the full historical dataset. Commodity SSDs are acceptable for full nodes, while validators should prioritize low-latency NVMe drives to minimize the small disk I/O component of block processing time.

💡 Compared to Other L1s

An Ethereum full node requires 2 TB+ of storage and continues growing by hundreds of GB per year. An Ethereum archive node requires 10–15 TB. By contrast, Sonic's SonicDB means a full node starts below 100 GB and grows at a fraction of the rate, making long-term node operation economically sustainable on standard hardware.

Installation

Installation & Configuration

Step-by-step guide to installing the Sonic client (go-opera), configuring genesis, and joining the mainnet.

Step 1: Install Go and Dependencies

# Update system packages
sudo apt update && sudo apt upgrade -y

# Install Go (1.21+)
wget https://go.dev/dl/go1.21.5.linux-amd64.tar.gz
sudo tar -C /usr/local -xzf go1.21.5.linux-amd64.tar.gz
echo 'export PATH=$PATH:/usr/local/go/bin' >> ~/.bashrc
source ~/.bashrc

# Install build dependencies
sudo apt install -y git make gcc build-essential

Step 2: Clone and Build go-opera

# Clone the Sonic client repository
git clone https://github.com/Fantom-foundation/go-opera.git
cd go-opera

# Checkout the latest stable release tag
git checkout $(git describe --tags --abbrev=0)

# Build the client
make opera

# Verify installation
./build/opera version

Step 3: Configure Genesis and Start Syncing

# Download the genesis file
wget https://download.soniclabs.com/mainnet/genesis/sonic-mainnet-genesis.g \
     -O ~/sonic/genesis.g

# Initialize the data directory
./build/opera --datadir ~/sonic/data \
              --genesis ~/sonic/genesis.g \
              --chain 146 \
              account new  # Create a local keystore account

# Start the node
./build/opera \
  --datadir ~/sonic/data \
  --genesis ~/sonic/genesis.g \
  --chain 146 \
  --http --http.addr 127.0.0.1 \
  --http.port 18545 \
  --http.api eth,net,web3 \
  --ws --ws.addr 127.0.0.1 \
  --ws.port 18546

Step 4: Set Up systemd Service

# /etc/systemd/system/sonic-node.service
[Unit]
Description=Sonic L1 Node
After=network.target

[Service]
User=sonic
ExecStart=/home/sonic/go-opera/build/opera \
  --datadir /home/sonic/sonic/data \
  --genesis /home/sonic/sonic/genesis.g \
  --chain 146 \
  --http --http.addr 127.0.0.1 \
  --http.port 18545 \
  --http.api eth,net,web3
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target
sudo systemctl daemon-reload
sudo systemctl enable sonic-node
sudo systemctl start sonic-node
sudo journalctl -u sonic-node -f  # Follow logs
Validation

Validator Setup

Participate in Lachesis aBFT consensus, earn staking rewards, and strengthen Sonic network security as a validator.

Validators are the backbone of Sonic's Lachesis aBFT consensus mechanism. By staking S tokens and running a validator node, you participate directly in the creation and finalization of event blocks — the core process by which the Sonic network achieves its sub-second finality. In return for this service, validators earn staking rewards proportional to their stake weight, plus a share of the transaction fees routed to them by the FeeM system.

Becoming a validator on Sonic requires meeting a minimum staking threshold of S tokens, which is set by governance and can be found in the current network parameters. The staking process involves locking S tokens in the on-chain staking contract, registering a validator key, and ensuring your node meets the uptime requirements. Validators that fail to maintain adequate uptime or that exhibit Byzantine behavior (signing conflicting blocks) are subject to slashing, which permanently reduces their staked balance as a penalty for harming the network.

Validator Registration Steps

  1. Generate validator keys

    Create a dedicated validator keypair separate from your funding wallet. Store the private key securely — on a hardware security module (HSM) for production validators. Never expose validator keys on internet-connected systems.

  2. Stake the minimum S tokens

    Send the required minimum S token amount to the SFC (Special Fee Contract) staking contract on Sonic mainnet. This locks your tokens and registers your validator entry. The staking transaction includes your validator public key.

  3. Configure and start the validator node

    Pass your validator key to the go-opera client using the --validator.id and --validator.pubkey flags. The node will immediately begin participating in consensus once synced and registered.

  4. Monitor validator performance

    Use the Sonic explorer's validator tab to monitor your validator's uptime, block production rate, and reward history. Set up alerting (PagerDuty, OpsGenie) on node health metrics to catch issues before they affect uptime SLAs.

⚠️ Slashing Conditions

Validators can be slashed for: double-signing event blocks (equivocation), being offline for extended periods below the minimum uptime threshold, or submitting invalid event blocks. Slashing permanently reduces the validator's staked balance. Run a robust monitoring setup and keep your client software up to date to minimize slashing risk.

Operations

Monitoring & Maintenance

Best practices for keeping your Sonic node healthy, up-to-date, and performing at its best.

Production Sonic node operation requires ongoing monitoring and maintenance. The go-opera client exposes a Prometheus-compatible metrics endpoint that provides detailed telemetry on node performance: block processing time, peer count, mempool depth, database read/write latency, event propagation time, and consensus round progress. Collecting these metrics into a Prometheus server and visualizing them with Grafana dashboards gives node operators full operational visibility.

Prometheus Configuration

# Start go-opera with metrics enabled
./build/opera \
  --datadir ~/sonic/data \
  --metrics \
  --metrics.addr 0.0.0.0 \
  --metrics.port 6060 \
  [other flags...]

# prometheus.yml — add Sonic job
scrape_configs:
  - job_name: 'sonic-node'
    static_configs:
      - targets: ['localhost:6060']
    scrape_interval: 15s

Software Upgrades

Sonic releases regular software updates to go-opera that include performance improvements, bug fixes, and protocol upgrades. Subscribe to the Sonic GitHub releases page and Discord announcements to receive timely notification of new releases. For mandatory protocol upgrades (hard forks), all validators must upgrade before the specified block height or they will fall out of consensus. Use a blue/green deployment strategy for validators — bring up a fully synced secondary node with the new version before switching over, to minimize downtime during upgrades.

Snapshot Sync

For operators setting up a new node, syncing from genesis can take significant time. Sonic provides official pruned state snapshots that allow new full nodes to sync to near-current state in hours rather than days. Download the latest snapshot, extract it to your data directory, and start go-opera — it will sync only the remaining blocks from the snapshot height to the current chain tip. Snapshots are published weekly and are cryptographically verified against the canonical chain state.

📊 Community Dashboards

The Sonic community maintains shared Grafana dashboard templates for go-opera node monitoring on the Sonic GitHub organization. Import these pre-built dashboards into your Grafana instance to get comprehensive node monitoring without building dashboards from scratch. Contribute improvements back to the community repository.
