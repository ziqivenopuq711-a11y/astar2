Run an Astar Node

Support the Astar Network by running a full node, archive node, or collator. Astar runs as a Polkadot parachain — collators produce blocks that are finalized by Polkadot relay chain validators. This guide covers everything from hardware selection to production monitoring.

Download Binary ↗ Architecture Docs
Overview

Node Types

Astar supports three node types, each serving a different purpose in the network's infrastructure.

🖥️

Full Node

A full node syncs and validates all recent blocks but does not store the entire history. Suitable for RPC endpoints, wallet backends, and dApp infrastructure. Pruned state keeps storage requirements manageable. Recommended for most operators.

RPC Provider Pruned State
🗄️

Archive Node

An archive node retains the full historical state of every block. Required for applications that query historical state (e.g., portfolio trackers, analytics, indexers like SubQuery or The Graph). Requires significantly more storage than a full node.

Full History Indexers
⛏️

Collator

Collators are Astar's block producers. They build parachain blocks and submit them to Polkadot relay chain validators for finalization. Collators do not need to hold a security bond but must maintain high uptime and performance. Rewards are paid in ASTR.

Block Producer ASTR Rewards
Parachain Architecture

As a Polkadot parachain, Astar's collators only need to produce candidate blocks — relay chain validators (secured by DOT staking) finalize them. This means Astar nodes do NOT need to run relay chain validation logic. However, collator nodes must also run a Polkadot relay chain node in parallel (either full or light client) to stay in sync with the relay chain state.

Full / RPC Node Minimums
  • CPU: 4-core x86-64 (Intel Xeon or AMD EPYC recommended)
  • RAM: 16 GB DDR4 minimum, 32 GB recommended
  • Storage: 500 GB NVMe SSD (pruned state grows ~50 GB/month)
  • Network: 100 Mbps symmetric, low-latency connection
Collator Node Requirements
  • CPU: 8+ cores, single-thread performance matters (3.5 GHz+)
  • RAM: 32 GB DDR4 or more
  • Storage: 1 TB NVMe SSD (collator + relay chain node)
  • Network: 1 Gbps dedicated, multiple provider redundancy
  • Uptime: 99.9%+ SLA target
Installation

Installation

Download the Astar binary and sync your node in four steps.

  1. Download the Astar Binary

    Get the latest release from the official GitHub repository. Always verify the SHA256 checksum before running.

    # Download latest astar-collator binary
LATEST=$(curl -s https://api.github.com/repos/AstarNetwork/Astar/releases/latest \
  | grep "tag_name" | cut -d'"' -f4)

wget -O astar-collator \
  "https://github.com/AstarNetwork/Astar/releases/download/${LATEST}/astar-collator-${LATEST}-ubuntu-x86_64.tar.gz"

# Or via cargo (builds from source)
git clone https://github.com/AstarNetwork/Astar
cd Astar
cargo build --release --features astar-native

  2. Configure Chain Spec

    Astar uses an on-chain specification. Download the chain spec for your target network:

    # Download Astar Mainnet chain spec
./astar-collator build-spec \
  --chain astar \
  --raw > astar-raw.json

# Shibuya Testnet chain spec
./astar-collator build-spec \
  --chain shibuya \
  --raw > shibuya-raw.json

  3. Start the Node

    Launch a full node with EVM RPC enabled. The node will begin syncing from the genesis block (or a snapshot if configured).

    ./astar-collator \
  --chain astar \
  --name "my-astar-node" \
  --rpc-port 9944 \
  --rpc-cors all \
  --rpc-methods Safe \
  --pruning archive \
  --state-cache-size 1073741824 \
  --base-path /var/lib/astar \
  -- \
  --chain polkadot \
  --sync fast

  4. Verify Sync Status

    Check that your node is syncing correctly by querying the local RPC endpoint:

    # Check sync status
curl -H "Content-Type: application/json" \
  -d '{"id":1,"jsonrpc":"2.0","method":"eth_blockNumber","params":[]}' \
  http://localhost:9944

# Compare with public RPC
curl -H "Content-Type: application/json" \
  -d '{"id":1,"jsonrpc":"2.0","method":"eth_blockNumber","params":[]}' \
  https://evm.astar.network
Operations

Monitoring & Maintenance

Production node operations: Prometheus metrics, Grafana dashboards, and staying current with runtime upgrades.

Prometheus & Grafana

Metrics Endpoint

Astar nodes expose Prometheus-compatible metrics on port 9615 by default. Start the node with --prometheus-port 9615 to enable.

# prometheus.yml scrape config
scrape_configs:
  - job_name: "astar_node"
    static_configs:
      - targets: ["localhost:9615"]
    scrape_interval: 15s

# Key metrics to monitor:
# substrate_block_height           — current sync height
# substrate_sub_libp2p_peers_count — peer count
# substrate_state_db_cache_bytes   — state cache usage
# substrate_tasks_spawned_total    — async task health

Use the Substrate Telemetry Grafana dashboard as a starting point. Import dashboard ID 13759 into your Grafana instance for pre-built Substrate node panels.

Runtime Upgrade Tracking

Stay Current with Upgrades

Astar uses Substrate's forkless runtime upgrade mechanism. When an upgrade is enacted on-chain, all nodes update automatically without manual intervention. However, binary upgrades (host function changes, storage migrations) require manual node binary updates.

  • Subscribe to GitHub Releases for binary upgrade notifications
  • Monitor the Astar Forum for runtime upgrade announcements
  • Join the #node-operators channel on Discord
  • Set up alerting when your node falls behind latest finalized block by more than 100 blocks
Systemd Service
# /etc/systemd/system/astar.service
[Unit]
Description=Astar Network Node
After=network.target

[Service]
User=astar
ExecStart=/usr/local/bin/astar-collator \
  --chain astar \
  --base-path /var/lib/astar
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
