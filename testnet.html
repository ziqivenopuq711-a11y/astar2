Overview

Blaze Testnet Overview

Sonic's Blaze testnet provides a production-equivalent environment for testing smart contracts, integrations, and front-end applications.

The Blaze testnet (Chain ID: 64165) is Sonic's primary developer testing network. Unlike many testnets that run simplified or downgraded versions of the mainnet protocol, Blaze mirrors Sonic mainnet's full technical capabilities: the same SonicVM execution engine, the same SonicDB state storage architecture, the same Lachesis aBFT consensus mechanism, and the same JSON-RPC API surface. What works on Blaze will work on mainnet — and what fails on Blaze will fail on mainnet, making Blaze an accurate predictor of production behavior.

Blaze testnet is designed to accommodate all types of development workflows. Solo developers can iterate quickly using free test S tokens from the faucet, deploying new contract versions as frequently as needed without cost concern. Teams can run end-to-end integration tests against a live network with realistic block times and finality behavior that local development environments cannot replicate. Protocols conducting security audits can point their auditors at Blaze deployments to test attack vectors in a realistic network environment.

⚠️ Testnet Disclaimer

Test S tokens on Blaze have no real-world monetary value and cannot be transferred to or from mainnet. The testnet may occasionally be reset or upgraded independently of mainnet. Always retest your application on Blaze after each testnet upgrade before deploying new versions to mainnet.

Adding Blaze to MetaMask

  1. Open MetaMask Network Settings

    Click the network selector at the top of MetaMask (it may say "Ethereum Mainnet"). Select "Add network" at the bottom of the dropdown list.

  2. Choose "Add a network manually"

    Click "Add a network manually" at the bottom of the network list to open the manual configuration form.

  3. Enter Blaze network details

    Fill in: Network Name → Sonic Blaze Testnet, RPC URL → https://rpc.blaze.soniclabs.com, Chain ID → 64165, Currency Symbol → S, Block Explorer URL → https://testnet.sonicscan.org.

  4. Save and switch

    Click "Save". MetaMask will validate the RPC endpoint and switch to Sonic Blaze Testnet. Your S balance will initially show 0 — use the faucet to obtain test tokens.

  5. Get test S tokens

    Visit faucet.soniclabs.com, connect your wallet or enter your address, and request test tokens. Tokens are sent within seconds.

Testing

Testing & Debugging on Blaze

Best practices for smart contract testing on the Sonic Blaze testnet — from unit tests to full integration test suites.

Effective testing on Blaze involves a layered approach. The first layer is unit testing with a local Hardhat or Foundry network — fast, isolated tests that run entirely in-process and provide instant feedback during development. The second layer is integration testing against Blaze, where your contracts interact with a real network that has realistic block times, mempool behavior, and finality characteristics. This two-layer approach catches different classes of bugs: unit tests find logic errors, while Blaze integration tests find timing, gas, and interaction issues that only manifest in a live network context.

Hardhat Testing Against Blaze

// test/MyContract.blaze.test.js
const { expect } = require("chai");
const { ethers } = require("hardhat");

// Run with: npx hardhat test --network sonicBlaze
describe("MyContract (Blaze integration)", function () {
  this.timeout(30000); // Allow time for network round-trips

  let contract;

  before(async function () {
    const [deployer] = await ethers.getSigners();
    const MyContract = await ethers.getContractFactory("MyContract");
    contract = await MyContract.deploy();
    await contract.waitForDeployment();
    console.log("Deployed to:", await contract.getAddress());
  });

  it("should process transactions with finality", async function () {
    const tx = await contract.myFunction(42);
    const receipt = await tx.wait(1); // Wait for 1 confirmation
    // On Sonic, this completes in under 1 second
    expect(receipt.status).to.equal(1);
  });
});

Gas Estimation and Optimization

Blaze uses the same gas pricing model as Sonic mainnet, making it an accurate environment for gas estimation. Run eth_estimateGas on Blaze to get reliable gas usage figures for your transactions, then use these to set appropriate gas limits in your deployment scripts and front-end applications. Sonic's SonicVM typically executes common EVM operations more efficiently than the reference Ethereum implementation, so gas usage on Sonic is often lower than on Ethereum for the same bytecode.

Debugging Failed Transactions

When a transaction reverts on Blaze, the testnet explorer displays the revert reason if the contract uses custom error types or require(condition, "reason string"). For more detailed debugging, use eth_call with the same transaction data to simulate the execution and obtain the full revert trace. Hardhat's --verbose flag and Foundry's forge test -vvvv verbosity levels provide stack traces that identify the exact line causing the revert in your Solidity source.

Blaze mirrors SonicVM's super instruction behavior, which means that performance characteristics you observe on Blaze accurately predict mainnet performance. If your contract runs efficiently on Blaze — processing complex operations within a single block, emitting the expected events, and consuming predictable amounts of gas — it will perform identically on mainnet under the same conditions.
