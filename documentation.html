Overview

Introduction to Sonic L1

The highest-performing EVM-compatible Layer-1 blockchain, purpose-built for next-generation decentralized finance.

Sonic is a groundbreaking Layer-1 blockchain that represents the natural evolution of the Fantom network. Built from first principles to overcome the throughput limitations that have long plagued smart-contract platforms, Sonic delivers a combination of extreme performance and full Ethereum Virtual Machine (EVM) compatibility that no other network has achieved at this scale.

At its core, Sonic is designed around a single philosophy: developers and users deserve a blockchain that is as fast, cheap, and reliable as the best centralized systems — without sacrificing decentralization or security. This translates into a network capable of processing up to 400,000 transactions per second, achieving finality in under one second, and offering developers a familiar EVM environment where existing Solidity contracts can be deployed without modification.

Sonic's peak throughput capacity dwarfs competing L1s. The network is benchmarked at 10,000 verifiable TPS in real-world conditions with standard validator hardware, giving applications the headroom to scale without network congestion.

Transactions on Sonic are final in under one second. Unlike probabilistic finality systems that require many block confirmations, Sonic's Lachesis aBFT consensus achieves deterministic finality immediately after the consensus round completes.

Full EVM Compatibility
Every Ethereum tool — Hardhat, Foundry, Remix, MetaMask, ethers.js, web3.js, viem — works on Sonic without modification. Deploy existing contracts directly to Sonic mainnet (Chain ID 146) with zero code changes required.

The developer-first design philosophy permeates every aspect of Sonic. From the Innovator Fund's 200 million S token allocation for ecosystem grants, to the FeeM system that lets builders monetize their on-chain activity, Sonic positions itself as the preferred home for serious DeFi projects seeking to offer users a premium experience without prohibitive gas costs.

📌 Key Facts

Sonic was launched as a successor to Fantom Opera. It maintains full state and account continuity for projects that migrated, while introducing an entirely new execution engine, storage layer, and incentive model. The native token is S, which is used for gas fees, staking, and governance.

Technical Deep-Dive

Architecture Overview

Three core innovations — SonicVM, SonicDB, and Lachesis aBFT — work together to deliver unprecedented performance.

🖥️ SonicVM — Optimized Execution Engine

SonicVM is Sonic's custom-built virtual machine designed to execute EVM bytecode significantly faster than the reference Ethereum implementation. It introduces super instructions — composite opcodes that combine multiple common EVM operations into a single, highly-optimized execution step. This eliminates the interpreter overhead associated with sequential opcode dispatch.

Super instructions work by profiling common opcode sequences across real-world smart contracts and compiling them into native machine code paths. The result is that complex DeFi operations — such as AMM swaps, lending calculations, and NFT transfers — execute in a fraction of the time they would take on a standard EVM. SonicVM remains fully backward-compatible: every valid EVM bytecode sequence produces the same output as it would on Ethereum, ensuring correctness and portability.

Developers do not need to write any special code to benefit from SonicVM. Existing Solidity or Vyper contracts compiled with standard toolchains automatically take advantage of the optimized execution paths when deployed to Sonic.

🗄️ SonicDB — Efficient State Storage

SonicDB is Sonic's redesigned state storage layer that achieves a 96% reduction in node operating costs compared to traditional blockchain node implementations. This dramatic improvement comes from a fundamental rethinking of how blockchain state is stored, indexed, and accessed on disk.

Traditional EVM nodes store the entire Merkle Patricia Trie (MPT) state history, which grows unboundedly over time and requires significant IOPS for state access. SonicDB replaces this with a purpose-built columnar storage format optimized for the access patterns of a live blockchain node. Frequently accessed state — current account balances, contract storage — is kept in fast-path caches, while historical state is archived efficiently.

The practical result for node operators is dramatically lower hardware requirements: SonicDB nodes can run comfortably on consumer-grade SSDs without the enterprise NVMe arrays required by comparable Ethereum archive nodes. This democratizes participation in the Sonic network and improves its decentralization.

🔗 Lachesis aBFT Consensus

Sonic inherits and refines the Lachesis asynchronous Byzantine Fault Tolerant (aBFT) consensus protocol from Fantom. Lachesis is a DAG-based (Directed Acyclic Graph) consensus mechanism that allows validators to create and gossip event blocks asynchronously, without the need for a designated leader or synchronous round timers.

In practical terms, aBFT means the network can reach consensus and finalize transactions even when some validators are offline or behave maliciously, as long as at least two-thirds of validators (by staked weight) are honest and responsive. Unlike traditional BFT protocols that are sensitive to network delays, Lachesis's asynchronous design allows it to operate efficiently across geographically distributed validator sets with varying latencies.

Incentive Model

FeeM — Fee Monetization

A revolutionary revenue model that lets developers earn up to 90% of the transaction fees generated by their dApps.

Fee Monetization (FeeM) is one of Sonic's most distinctive features and a fundamental departure from how all other EVM-compatible blockchains handle transaction fee distribution. On Ethereum and most competing L1s, 100% of the base fee is either burned (EIP-1559) or distributed to validators — application developers receive nothing from the on-chain activity they drive.

Sonic's FeeM system changes this equation entirely. When a user submits a transaction that interacts with a registered dApp contract, up to 90% of the gas fees paid for that transaction are rebated directly to the contract developer's registered address. The remaining portion goes to validators, ensuring network security remains well-funded. This creates a sustainable, Web2-like recurring revenue model for Web3 application developers.

💰 How FeeM Works

Register your deployed contract address on Sonic's FeeM portal. Once registered, all gas fees generated by interactions with your contract are split: 90% is returned to your registered wallet in S tokens, and 10% goes to validators. Payouts are automatic and occur on-chain.

📈 Aligned Incentives

FeeM directly aligns the incentives of developers, users, and the network. Developers are incentivized to attract high-volume usage because more transactions mean more revenue. Users benefit because popular, revenue-generating dApps can afford to subsidize gas or invest in better UX.

The FeeM model makes Sonic uniquely positioned for DeFi protocols, DEXs, and high-volume applications. A DEX processing 1,000,000 swaps per month at 0.1 gwei average gas cost earns substantial S token revenue purely from existing user activity — no token sale, no VC funding required to sustain operations. This is the sustainable economics that Web3 has been missing.

Registration for FeeM is permissionless and on-chain. Any developer who has deployed a smart contract to Sonic mainnet can register for FeeM. Once registered, fee rebates begin immediately on the next qualifying transaction. There is no minimum contract size, no whitelist, and no approval process — FeeM is available to every builder on Sonic equally.

Connecting with ethers.js v6

import { ethers } from "ethers";

// Connect to Sonic mainnet
const provider = new ethers.JsonRpcProvider("https://rpc.soniclabs.com");

// Check chain ID
const network = await provider.getNetwork();
console.log("Chain ID:", network.chainId); // 146n

// Query latest block
const block = await provider.getBlockNumber();
console.log("Latest block:", block);

// Check S token balance
const balance = await provider.getBalance("0xYourAddress");
console.log("Balance:", ethers.formatEther(balance), "S");

Connecting with web3.js

import Web3 from "web3";

const web3 = new Web3("https://rpc.soniclabs.com");

// Verify connection
const chainId = await web3.eth.getChainId();
console.log("Chain ID:", chainId); // 146

// Get gas price
const gasPrice = await web3.eth.getGasPrice();
console.log("Gas price (wei):", gasPrice);

  1. Install dependencies

    Run npm install ethers or npm install web3 in your project directory. Both libraries work identically on Sonic as on Ethereum.

  2. Add Sonic to MetaMask

    Open MetaMask → Settings → Networks → Add Network. Enter Chain ID 146, RPC URL https://rpc.soniclabs.com, and symbol S.

  3. Get test tokens

    Use the Blaze Testnet (Chain ID 64165) and the Sonic faucet to obtain test S tokens for development. See the Testnet page for instructions.

  4. Deploy your contract

    Configure Hardhat or Foundry with the Sonic RPC endpoint. Compile and deploy as you would on any EVM-compatible network. See Development Tools for full configuration examples.
